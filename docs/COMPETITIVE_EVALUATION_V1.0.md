# zt-gateway 競合比較・差別化検証シート (v1.0)

## 目的

`zt-gateway` の差別化仮説を、競合比較と実装現実性の両面で評価し、`v0.3` / `v0.5` / `v1.0` の優先順位に落とし込む。

## 比較対象

- zt-gateway (独自構想)
- Wormhole (即時共有)
- Taildrop (自端末間)
- OnionShare (匿名性)
- Proton / Tresorit (管理重視)
- Bitwarden Send (軽量共有)

## 提案アイデアの評価

### A. 既存操作に溶け込むステルス・セキュリティ (Invisible UX)

- 刺さり度: 高
- 実装難易度: 中
- リスク: OS統合ごとの差分吸収
- 推奨: `v0.3` から着手

### B. 受信側ゼロインストール検証 (Zero-Install Verification)

- 刺さり度: 非常に高
- 実装難易度: 高
- リスク: ブラウザでの検証体験と署名検証の整合性設計
- 推奨: `v0.5` でPoC、`v1.x` で本格化

### C. コンテキスト・アウェア共有 (Smart File Contract)

- 刺さり度: 高 (エンタープライズに強い)
- 実装難易度: 高
- リスク: 誤判定時の業務停止、ポリシー運用複雑化
- 推奨: `v1.0` で基本条件のみ、詳細条件は `v1.x`

### D. クラウド寄生型セキュリティ (Cloud-Agnostic Wrapper)

- 刺さり度: 非常に高
- 実装難易度: 中
- リスク: 各クラウド導線のAPI/UX変更追従
- 推奨: `v0.5` の主戦略

### E. デバイス拘束 + 時限失効 (Device-Bound Ephemeral Files)

- 刺さり度: 高 (高機密市場)
- 実装難易度: 非常に高
- リスク: 端末識別の信頼性、誤失効による業務インパクト
- 推奨: `v1.x` の上位機能として段階導入

### F. トラスト・ツリー・ダッシュボード (Blockchain-Ready Provenance)

- 刺さり度: 中-高 (証跡重視市場)
- 実装難易度: 中-高
- リスク: 先に監査ログ基盤を固めないと形骸化
- 推奨: `v0.5` で可視化、ブロックチェーン打刻は `v1.x`

## Codex提案 (追加差別化案)

### 1) Verification Receipt (検証レシート) を標準化

- 概要: 受信検証の結果を `receipt.json` として機械可読で出力し、Slack/チケットにそのまま添付できるようにする
- 刺さる理由: 「検証した」ではなく「検証証拠を提出できる」ため、監査/取引先対応で強い
- 最短実装: `zt verify --receipt-out <path>` と `--share-json` へのreceipt参照追加

### 2) Trust Profiles (安全レベルを1フラグ化)

- 概要: `--profile public|internal|confidential|regulated` のように、安全設定一式をプリセット化する
- 刺さる理由: ポリシー詳細を知らない利用者でも、業務区分で安全運用できる
- 最短実装: 既存policyをプロファイル名にマップし、CLI表示に「現在プロファイル」を明記

### 3) Friction Budget SLO (UX運用品質の数値管理)

- 概要: 「初回成功率」「再試行率」「復旧所要時間」をSLOとして毎週可視化する
- 刺さる理由: 無意識UXを感覚ではなく運用品質として改善できる
- 最短実装: `zt send/verify` 実行時に匿名メトリクスをローカル集計し、週次レポート化

### 4) Safe Failure Envelope (失敗時の安全封筒)

- 概要: 失敗時に、原因・再実行コマンド・runbookリンクを1つの再実行可能テキストで出力する
- 刺さる理由: Helpdesk依存を減らし、失敗から自己復旧までの時間を短縮する
- 最短実装: 現在のエラーコード出力に `quick_fix_bundle` を追加

### 5) Sidecar Mode (既存CLIへの薄い寄生)

- 概要: `cp/scp` 互換の薄いラッパーモードを提供し、既存運用の置換コストを最小化する
- 刺さる理由: 「新しい操作を覚える負担」を抑えて導入障壁を下げられる
- 最短実装: `zt cp ...` を実装し、内部で `scan -> sanitize -> pack -> share` を実行

## 必勝パターン (結論)

`zt-gateway` が勝つ形は次の3点の同時成立。

1. 既存クラウドの利便性を捨てない (寄生・無意識UX)
2. 監査証跡とポリシー制御を標準装備する
3. 受信側は最小摩擦で安全状態を確認できる (1行判定 + 将来のブラウザ完結)

競合が「送れる」または「保管できる」に寄る中で、`zt-gateway` は
**「送れる + 検証できる + 証跡が残る」を普段の操作で成立させる** ことが差別化の核心。
