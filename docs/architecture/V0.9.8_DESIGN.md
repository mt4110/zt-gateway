# v0.9.8 Design: Dashboard Mutating API Fail-Closed Hardening

## 0. Baseline

- 確認日: 2026-02-27
- ブランチ: `feat/v0.9.7`
- 現状:
  - `v0.9.7` で `dashboard_alert_dispatch_unsafe_config` 可視化と専用ゲートは導入済み
  - dashboard の listen address は `--addr` で任意指定可能（default は `127.0.0.1:8787`）
  - mutating endpoint（`/api/lock`, `/api/incident`, `/api/alerts/dispatch`, `/api/key-repair/jobs/*/transition`）に認可境界がない
- 残ギャップ:
  - `--addr 0.0.0.0:8787` など非 loopback bind でも mutating endpoint が同じ挙動で公開される
  - 「remote bind + mutation auth 未設定」を danger/alerts に deterministic に出せない
  - 当該回帰を独立検知する `v0.9.8` 専用ゲートが未定義

## 1. Goal / Non-Goal

### Goal

- v1.0 の運用安全ゴール（誤公開しても破壊操作は通らない）を先に固定し、`v0.9.8` では dashboard mutation 面を fail-closed にする
- remote bind 時に mutation auth 未設定なら「read-only へ自動降格」し、拒否理由を機械可読で返す
- danger/alerts/契約テスト/専用CI gate を一体で固定し、回帰を単独検知できる状態にする

### Non-Goal

- SSO/OIDC やロールベース認可（RBAC）の導入
- dashboard UI デザイン刷新
- Control Plane API 仕様変更

## 2. Goal からの逆算

### 最終到達状態（v1.0運用要件）

- ネットワーク露出を誤っても、未認証の mutation は通らない
- 拒否時の理由が `error_code` と監査イベントで追跡可能
- CI で「unsafe 公開の再発」を即時検知できる

### v0.9.8 で閉じる最小成立条件

- 非 loopback bind かつ mutation token 未設定では mutating endpoint を必ず拒否（read-only fail-closed）
- token 設定時はヘッダ一致でのみ mutation を許可
- unsafe 状態を dashboard snapshot の danger/alerts へ deterministic に露出
- `check-v098-dashboard-auth-gate.sh` を pre-push に組み込み

## 3. Contract Scope (C-9801..C-9804)

### C-9801 Remote Bind Mutation Lock Contract

- 条件:
  - dashboard listen address が非 loopback（`127.0.0.1` / `::1` / `localhost` 以外）
  - `ZT_DASHBOARD_MUTATION_TOKEN` が空
- 結果:
  - mutating endpoint は 403 で拒否
  - エラーコードは `dashboard_mutation_token_required`
  - dashboard snapshot に `danger.signals[].code=dashboard_remote_bind_mutation_locked` を追加（`level=high`）

### C-9802 Mutation Auth Header Contract

- `ZT_DASHBOARD_MUTATION_TOKEN` が設定されている場合:
  - mutating endpoint は `X-ZT-Dashboard-Token` 一致時のみ許可
  - 不一致/未指定は 403 `dashboard_mutation_auth_failed`
- 対象 endpoint:
  - `/api/lock`
  - `/api/incident`
  - `/api/alerts/dispatch`
  - `/api/key-repair/jobs/*/transition`

### C-9803 Alert Dispatch Audit Reason Contract

- `/api/alerts/dispatch` の監査イベント `reason_code` に auth 理由を固定追加:
  - `dashboard_mutation_token_required`
  - `dashboard_mutation_auth_failed`
  - （成功系既存）`dispatch_dry_run` / `dispatch_success`
- これにより通知失敗の一次切り分けを `reason_code` だけで再現可能にする

### C-9804 v0.9.8 Dedicated Gate Contract

- `scripts/ci/check-v098-dashboard-auth-gate.sh` を追加し、`C-9801..C-9803` の契約のみを単体実行
- `check-pre-push-readiness.sh` へ組み込み、dashboard mutation auth 回帰を独立検知

## 4. Implementation Tickets

1. TKT-9801 dashboard listen address の loopback/non-loopback 判定ユーティリティ追加
2. TKT-9802 mutating endpoint 共通 auth ガード（token required / mismatch）
3. TKT-9803 danger/alerts へ `dashboard_remote_bind_mutation_locked` 追加
4. TKT-9804 `/api/alerts/dispatch` の auth reason_code 監査契約追加
5. TKT-9805 契約テスト追加（remote no-token reject / token mismatch / token match）
6. TKT-9806 `check-v098-dashboard-auth-gate.sh` 追加 + pre-push 統合
7. TKT-9807 README / OPERATIONS / CHANGELOG 更新

## 5. Exit Criteria

- 非 loopback bind + token未設定で mutating endpoint が常に `dashboard_mutation_token_required` を返す
- token不一致で `dashboard_mutation_auth_failed`、一致で mutation 成功
- unsafe 状態で `dashboard_remote_bind_mutation_locked` が snapshot に必ず出る
- `v0.9.8` 専用ゲートが pre-push に統合される
