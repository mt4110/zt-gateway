# v0.9.9 Design: Dashboard Mutation Auth Coverage Closure

## 0. Baseline

- 確認日: 2026-02-27
- ブランチ: `feat/v0.9.7`
- 現状:
  - v0.9.8 で dashboard mutation auth（remote bind時 fail-closed + token一致必須）を導入済み
  - `/api/lock` `/api/incident` `/api/alerts/dispatch` `/api/key-repair/jobs/{job_id}/transition` は認可ガード済み
- 残ギャップ:
  - mutation endpoint である `/api/keys/{key_id}/status` と `POST /api/key-repair/jobs` が認可ガード対象から漏れている
  - remote bind かつ token未設定時でも当該2 endpoint が実行可能となる
  - v0.9.9 専用の回帰検知ゲートが未定義

## 1. Goal / Non-Goal

### Goal

- dashboard mutation endpoint の auth coverage を 100% にする
- remote bind時の fail-closed と token認証契約を、全 mutation endpoint で同一挙動に統一する
- v0.9.9 専用ゲートで回帰を独立検知できる状態にする

### Non-Goal

- 認証方式の変更（OIDC/RBAC/step-up導入）
- dashboard 読み取り系 API の仕様変更
- Control Plane API の変更

## 2. Goal からの逆算

### 最終到達状態（v1.0運用要件）

- dashboard が誤って非 loopback 公開されても、すべての mutation API が未認証操作を拒否する
- 拒否理由は `dashboard_mutation_token_required` / `dashboard_mutation_auth_failed` に固定され、一次切り分けが deterministic
- CI で auth coverage 漏れを単独検知できる

### v0.9.9 で閉じる最小成立条件

- `/api/keys/{key_id}/status` に v0.9.8 と同じ mutation auth を適用
- `POST /api/key-repair/jobs` に v0.9.8 と同じ mutation auth を適用
- 未設定token/不一致token/一致token の 3ケース契約テストを追加
- `check-v099-dashboard-mutation-coverage-gate.sh` を pre-push に組み込む

## 3. Contract Scope (C-9901..C-9903)

### C-9901 Keys Status Mutation Auth Contract

- 対象: `POST /api/keys/{key_id}/status`
- 条件/結果:
  - remote bind + token未設定 -> 403 `dashboard_mutation_token_required`
  - token設定 + ヘッダ不一致 -> 403 `dashboard_mutation_auth_failed`
  - token設定 + ヘッダ一致 -> 200 で状態遷移を許可

### C-9902 Key Repair Create Mutation Auth Contract

- 対象: `POST /api/key-repair/jobs`
- 条件/結果:
  - remote bind + token未設定 -> 403 `dashboard_mutation_token_required`
  - token設定 + ヘッダ不一致 -> 403 `dashboard_mutation_auth_failed`
  - token設定 + ヘッダ一致 -> 201/200 で起票を許可

### C-9903 v0.9.9 Dedicated Gate Contract

- `scripts/ci/check-v099-dashboard-mutation-coverage-gate.sh` を追加し、v0.9.9 契約のみ単体実行できる
- `check-pre-push-readiness.sh` に組み込み、coverage 漏れを独立検知する

## 4. Implementation Tickets

1. TKT-9901 `handleDashboardKeyStatusAPI` に mutation auth 強制を追加
2. TKT-9902 `handleDashboardKeyRepairJobsCreate` に mutation auth 強制を追加

## 5. Exit Criteria

- `POST /api/keys/{key_id}/status` が v0.9.8 契約どおり fail-closed で認可される
- `POST /api/key-repair/jobs` が v0.9.8 契約どおり fail-closed で認可される
- v0.9.9 専用ゲートが pre-push へ統合される
