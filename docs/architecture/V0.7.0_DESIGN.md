## 1. Context

v0.6.1 で「同期・契約・可観測性」の土台は揃った。一方で README の既知ギャップとして以下が残っている。

- file type guard の対応形式/診断粒度が限定的で、block 理由の機械判定が弱い
- strict/degraded scan posture が profile と運用ルールで一貫して見えにくい
- `zt policy status --json` / `zt sync --json` は使えるが、一次対応の分岐が人間依存になりやすい

v0.7.0 は「運用で迷わない fail-closed」を狙い、policy/sync の診断契約を一段厳密化する。

## 2. Goal / Non-goal

### Goal

- file type block 理由を JSON 契約で追跡可能にする
- scan posture（strict/degraded許容）を profile 単位で機械判定可能にする
- policy/sync JSON を runbook アンカーと 1:1 で連動させる
- pre-push で v0.7.0 契約回帰を独立ゲートで検知できる

### Non-goal

- 新しい暗号アルゴリズム導入
- UI ダッシュボード刷新
- マルチリージョン同期

## 3. Scope (v0.7.0)

### v0.7.0-1 File Type Guard Explainability Contract

`zt send`/`zt scan` の file type 判定を機械可読で残す。

- `policy_magic_mismatch` 時に診断オブジェクトを固定出力
  - `extension`
  - `detected_kind`
  - `detected_mime`
  - `reason_code`（例: `expected_pdf`, `expected_text_like`, `pdf_missing_eof_marker`）
- policy decision JSON / event payload に同じ構造を持たせる
- 文字コード系（UTF-8/Shift-JIS）と OOXML 判定の contract test を増強

### v0.7.0-2 Scan Posture Contract

strict/degraded 許容を profile 起点で固定する。

- `scan policy` の有効 posture を JSON で明示
  - `strict_effective` (bool)
  - `required_scanners` ([]string)
  - `require_clamav_db` (bool)
  - `allow_degraded_scan` (bool)
- `public` 以外で degraded 実行が混入した場合は fail-closed
  - `sync_error_code` / `policy_decision.error_code` に `policy_scan_posture_violation` を追加

### v0.7.0-3 Aggregated Policy Status Contract

`zt policy status --json` に全体診断モードを追加する。

- `--kind all` を追加し extension/scan を同時出力
- 集約フィールドを追加
  - `overall_set_consistency`
  - `overall_freshness_state`
  - `critical_kinds` (`extension` / `scan`)
  - `set_consistency_reason`（`skew_detected`, `missing_extension`, `missing_scan`, `set_id_missing`）

### v0.7.0-4 Sync Backlog SLO Determinism Contract

backlog SLO の breach 状態を再現可能にする。

- `zt sync --json` に SLO コンテキストを追加
  - `backlog_slo_seconds`
  - `backlog_breached` (bool)
  - `backlog_breached_since` (RFC3339, optional)
- `sync_backlog_slo_breached` 時の quick fix を runbook アンカーで固定

### v0.7.0-5 Runbook Anchor Contract

`quick_fix_bundle` と `docs/OPERATIONS.md` の対応を固定する。

- `quick_fix_bundle` に `runbook_anchor` を追加
- 主要失敗コードで anchor を必須化
  - `policy_set_skew_detected`
  - `policy_sync_slo_breached`
  - `policy_scan_posture_violation`
  - `sync_backlog_slo_breached`
  - `ingest_ack_mismatch`

## 5. API Delta (Draft)

互換方針: 既存フィールドは維持し、追加のみで拡張する。

### 5.1 `GET /v1/policies/{kind}/latest` (`PolicyResponse`)

追加フィールド:

- `policy_kind` (`extension` | `scan`)
- `scan_requirements` (object, optional)
  - `required_scanners`
  - `require_clamav_db`
  - `allow_degraded_scan`
- `file_type_guard` (object, optional)
  - `enforced` (bool)
  - `contract_version` (string)

意図:

- CP が署名bundle配布時に運用 posture を top-level で可視化
- Gateway/Support が TOML 生パースなしで一次判定可能

### 5.2 `POST /v1/events/{scan|artifact|verify}` 202 (`IngestAccepted`)

追加フィールド:

- `ack_schema_version` (integer)
- `canonical_payload_sha256` (string)

意図:

- envelope/raw payload 正規化のズレを ACK 側で診断可能にする
- `ingest_ack_mismatch` の調査を短縮する

### 5.3 CLI JSON Contract Additions

- `zt policy status --json --kind all` の response スキーマ追加
- `zt sync --json` の backlog SLO コンテキスト追加
- `quick_fix_bundle.runbook_anchor` 追加

### 5.4 OpenAPI / Sync対象

- `docs/openapi/control-plane-v1.yaml`
- `control-plane/api/cmd/sync-openapi-enums/main.go`

## 7. Exit Criteria

- `zt send` / `zt scan` の block ケースで `file_type_guard.reason_code` が 100% 出力される
- `zt policy status --json --kind all` で extension/scan の集約診断が常時取得できる
- `zt sync --json` で backlog SLO 文脈（閾値/ breach 状態）が欠落しない
- `quick_fix_bundle.runbook_anchor` が主要失敗コードで必須になっている
- v0.7.0 追加 CI ゲートが 10 営業日連続で green

## 8. Implementation Order (Recommended)

1. Gateway contract test 先行追加（C-7001, C-7002, C-7003, C-7004, C-7005）
2. Control Plane API response 拡張（PolicyResponse / IngestAccepted）
3. Gateway JSON 出力拡張（policy status all / sync backlog SLO / runbook anchor）
4. OpenAPI 更新 + enum sync
5. CI gate 追加
6. README / OPERATIONS 更新

## 9. README 更新案（貼り付け用）

挿入位置案: 「現時点での安全な運用ガイド（推奨）」の v0.6.0MAX 箇条書きの直後。

追記案:

- v0.7.0 では file type block 時に `file_type_guard.reason_code` を JSON で確認し、拡張子偽装/形式不一致の一次判定を固定する
- v0.7.0 では `zt policy status --json --kind all` を一次確認コマンドにし、`overall_set_consistency` / `overall_freshness_state` を監視する
- v0.7.0 では `zt sync --json` の `backlog_breached` / `backlog_breached_since` を監視し、SLO breach 時は runbook アンカーへ直行する
- v0.7.0 では `quick_fix_bundle.runbook_anchor` を利用し、`error_code -> anchor` の運用を固定する
- 設計正本は `docs/architecture/V0.7.0_DESIGN.md`

```bash
go run ./gateway/zt policy status --json --kind all
go run ./gateway/zt sync --json
go run ./gateway/zt send --client <recipient> --share-json ./safe.txt
```
