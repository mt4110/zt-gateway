## 2. 残ギャップ棚卸し（True Zero Trust 観点）

### G-9101: degraded scan が設計上まだ許容される

- `profile=public` では strict が既定で無効化される
- `--allow-degraded-scan` で no-scanner-available を通せる
- True Zero Trust では「明示的 break-glass 以外は degraded 不可」に寄せる必要がある

証跡:
- `gateway/zt/commands_flow.go` の strict 判定ロジック
- `policy/profiles/public/scan_policy.toml` が `required_scanners=[]` / `require_clamav_db=false`

### G-9102: internal プロファイルだけ policy ファイル欠落時に built-in fallback する

- 欠落時 WARN + 既定値で継続するため、ポリシーの外部監査可能性が弱くなる
- True Zero Trust では「ポリシー成果物の明示配置」を必須化したい

証跡:
- `gateway/zt/commands_flow.go` 内の `os.IsNotExist(...) && profile=internal` fallback

### G-9103: verify の signer trust anchor が不十分

- `secure-pack` verify は「ローカル keyring 上で検証成功」を信頼している
- signer allowlist の強制は TODO のまま

証跡:
- `tools/secure-pack/internal/pack/unpack.go` の TODO (`Check if signer is in ALLOWLIST?`)

### G-9104: receipt provenance の鍵指紋が実検証結果に結びついていない

- receipt の `key_fingerprint` は ENV から推定し、無ければ `000...` を返す
- 監査証跡としては「実際に検証した署名者」を記録したい

証跡:
- `gateway/zt/verify_receipt.go` の `resolveReceiptKeyFingerprint()`

### G-9105: 「再構成必須」強制の対象が限定的

- README / THREAT_MODEL で未完了ギャップとして明示済み
- 現状は画像中心で、PDF/OOXML は scan-only 側が残る

証跡:
- `README.md` 「現在の穴・未完了の点」
- `THREAT_MODEL.md` 「Threats Not Fully Mitigated Yet」

### G-9106: actual gate の pin 配布運用が手作業依存

- CI 変数未設定で fail-closed するのは正しい
- ただし導入プロセスの標準化が不足し、運用で詰まりやすい

証跡:
- `scripts/ci/check-zt-setup-json-actual-gate.sh`
- `scripts/ci/check-pre-push-readiness.sh` step 14

## 3. v0.9.1 スコープ（提案）

### P0（True Zero Trust 最低条件）

1. **C-9101 Strict-Only Default Contract**
- `zt send` は全 profile で strict default
- degraded 許可は break-glass 明示時のみ

2. **C-9102 Explicit Policy Artifact Contract**
- `policy/*.toml` 欠落時は profile を問わず fail-closed
- built-in fallback を常用経路から除外

3. **C-9103 Signer Allowlist Verify Contract**
- verify 成功条件に signer fingerprint allowlist 一致を追加
- allowlist 未設定時は fail-closed（または明示 break-glass のみ）

4. **C-9104 Receipt Provenance Binding Contract**
- receipt に記録する fingerprint は verify 実測値のみ
- `000...` fallback を廃止

### P1（v1.0 手前で揃える）

5. **C-9105 Rebuild-Or-Deny Expansion Contract**
- confidential/regulated で PDF/OOXML を `SCAN_REBUILD` へ移行
- 未対応形式は fail-closed deny を明示

6. **C-9106 Pin Distribution Automation Contract**
- expected pin 配布を CI bootstrap 手順として固定
- ローカル/CI で同一診断結果になるようテンプレ化

## 4. 実装チケット案

## TKT-9101: strict default hardening

- 変更対象:
  - `gateway/zt/commands_flow.go`
  - `gateway/zt/cli_parse.go`
  - `gateway/zt/cli_parse_test.go`
  - `gateway/zt/policy_decision_contract_test.go`
- DoD:
  - `profile=public` でも strict default
  - degraded は明示 break-glass でのみ許可

## TKT-9102: policy artifact mandatory

- 変更対象:
  - `gateway/zt/commands_flow.go`
  - `gateway/zt/setup_preflight.go`
  - `gateway/zt/setup_contract_test.go`
- DoD:
  - internal でも policy 欠落時 fail-closed
  - `zt setup --json` に「欠落=fail」を固定

## TKT-9103: signer allowlist enforcement

- 変更対象:
  - `tools/secure-pack/internal/pack/unpack.go`
  - `tools/secure-pack/internal/gpg/*`（署名者 fingerprint 抽出）
  - `gateway/zt/commands_verify.go`
- DoD:
  - verify 成功時に signer fingerprint を確定取得
  - allowlist 不一致で verify fail

## TKT-9104: receipt provenance binding

- 変更対象:
  - `gateway/zt/verify_receipt.go`
  - `gateway/zt/commands_verify.go`
  - `gateway/zt/verify_receipt_test.go`
- DoD:
  - receipt の `key_fingerprint` は verify 実測値
  - unknown/zero fallback を廃止

## TKT-9105: rebuild-or-deny expansion

- 変更対象:
  - `policy/profiles/confidential/extension_policy.toml`
  - `policy/profiles/regulated/extension_policy.toml`
  - `gateway/zt/ext_policy.go`
  - `gateway/zt/commands_flow_test.go`
- DoD:
  - high-risk document の扱いが profile ごとに明示
  - 未対応再構成は fail-closed

## TKT-9106: pin distribution bootstrap

- 変更対象:
  - `scripts/dev/bootstrap-ci-root-pin-expected.sh`
  - `scripts/ci/check-pre-push-readiness.sh`
  - `docs/OPERATIONS.md`
- DoD:
  - expected-pin 経路が標準手順として確立
  - actual gate 停止理由の一次切り分けが 1 回で完了

## 5. 実装順序（推奨）

1. TKT-9102 policy artifact mandatory
2. TKT-9101 strict default hardening
3. TKT-9103 signer allowlist enforcement
4. TKT-9104 receipt provenance binding
5. TKT-9106 pin distribution bootstrap
6. TKT-9105 rebuild-or-deny expansion

## 6. Exit Criteria for feat/v0.9.1

- pre-push readiness が `[15/15]` まで通る（actual gate は expected pin bootstrap 前提）
- degraded 許可経路が break-glass に限定される
- verify の trust anchor が allowlist で機械検証される
- receipt provenance が実検証値と一致し、監査で再現可能
- README / THREAT_MODEL / OPERATIONS の既知ギャップ欄が v0.9.1 状態に更新される

## 7. Deferred Backlog (v0.9.2 Candidate)

### TKT-9201: Team/Enterprise Boundary Mode

- 背景:
  - 社内・チーム内限定でのみ利用できる運用モードを製品機能として固定したい
  - `v1.0.0` までに実運用可能な enterprise 境界制御を成立させる
- 方向性:
  - 受信者/署名者/共有経路を tenant/team 単位で fail-closed 制限
  - setup/doctor で境界設定の欠落・逸脱を deterministic に検知
  - share-json と verify/receipt に境界メタデータを固定出力
- 対応設計:
  - `docs/architecture/V0.9.2_DESIGN.md` を参照
