## 1. Context

v0.5g までで policy 配布の基礎契約（署名 bundle / keyset / canary / sync loop / min version fail-closed）は整っている。  
v0.6.0MAX は、現場運用での迷いを減らすために以下を契約化する段階とする。

- extension/scan の 2系統 policy を同時に運用しても整合が崩れない
- sync backlog と freshness を JSON で一意に判断できる
- Control Plane ingest 応答を監査可能にし、再送判定を機械化できる
- README と運用 runbook が同じ判定軸（error_code / freshness / backlog）で動く

## 2. Goal / Non-goal

### Goal

- 利用者が `zt setup --json` / `zt policy status --json` / `zt sync --json` だけで一次切り分け可能
- API 契約と CLI 契約が 1:1 で追跡可能（OpenAPI + contract test + CI gate）
- pre-push で v0.6.0MAX 契約回帰を自動検知できる

### Non-goal

- UI ダッシュボードの大規模刷新
- 新しい policy 言語設計
- マルチリージョン同期や分散合意

## 3. Scope (v0.6.0MAX)

### v0.6.0MAX-1 Policy Set Coherency Contract

`extension` と `scan` を別々に配布しても、同一 rollout 由来であることを追跡可能にする。

- policy bundle に `policy_set_id` を追加
- Gateway は active/staged/LKG それぞれに `policy_set_id` を保存
- `zt policy status --json` に `set_consistency` を追加
  - `consistent` | `skew_detected` | `unknown`
- skew 検知時は `sync_error_code=policy_set_skew_detected` を固定

契約テスト:

- 同一 `policy_set_id` で `consistent`
- extension/scan の `policy_set_id` 不一致で `skew_detected`
- 片系統未取得時は `unknown`（fail はしない）

### v0.6.0MAX-2 Policy Freshness SLO Contract

policy が古い状態を可視化し、profile ごとに運用判定を固定する。

- `zt policy status --json` に freshness 指標を追加
  - `last_sync_age_seconds`
  - `freshness_slo_seconds`
  - `freshness_state` (`fresh` | `stale` | `critical`)
- profile ごとの既定 SLO
  - `public/internal`: 24h
  - `confidential/regulated`: 6h
- `critical` のとき `sync_error_code=policy_sync_slo_breached`

契約テスト:

- SLO 内は `fresh`
- SLO 超過は `stale`
- 2x 超過は `critical`

### v0.6.0MAX-3 Ingest ACK Integrity Contract

event sync の再送可否を payload 単位で機械判定可能にする。

- `/v1/events/*` の `202` 応答（`IngestAccepted`）に以下を必須追加
  - `endpoint`
  - `payload_sha256`
  - `accepted_at` (RFC3339)
- Gateway 側は ACK の `payload_sha256` が送信値と一致することを検証
- 不一致時は `error_class=internal`, `error_code=ingest_ack_mismatch`

契約テスト:

- ACK echo 一致で正常
- `payload_sha256` 不一致で fail
- duplicate 応答でも hash 一致なら再送しない

### v0.6.0MAX-4 Sync Backlog Visibility Contract

spool 詰まりを CLI だけで判定可能にする。

- `zt sync --json` に backlog 指標を追加
  - `pending_count`
  - `oldest_pending_age_seconds`
  - `retryable_count`
  - `fail_closed_count`
  - `next_retry_at`
- `oldest_pending_age_seconds` が閾値超過時:
  - `error_class=retryable`
  - `error_code=sync_backlog_slo_breached`

契約テスト:

- pending 0 で age=0
- retryable と fail_closed が正しく分類
- backlog 閾値超過で `sync_backlog_slo_breached`

### v0.6.0MAX-5 Runbook-Linked Error Contract

CLI 失敗 JSON の運用導線を一本化する。

- `quick_fix_bundle` 必須項目を共通化
  - `why`
  - `commands[]`
  - `runbook`
  - `retry`
- policy/sync/setup の JSON 失敗で `quick_fix_bundle.runbook` を必須化
- runbook URL は `docs/OPERATIONS.md` に固定

契約テスト:

- 主要失敗コードで `quick_fix_bundle` が欠けない
- `runbook` が空文字にならない

## 5. API Delta (Draft)

互換方針: 既存フィールドを壊さず追加のみで拡張する。

### 5.1 `GET /v1/policies/{kind}/latest`

追加（response）:

- `policy_set_id` (string)
- `freshness_slo_seconds` (integer)

互換ルール:

- 既存クライアントは unknown field 無視で動作継続
- Gateway v0.6.0MAX は `policy_set_id` 欠落時に `set_consistency=unknown`

### 5.2 `GET /v1/policies/keyset`

追加（response）:

- `rotation_id` (string)
- `active_key_id` (string)
- `next_key_id` (string, optional)

利用目的:

- 運用側が keyset 状態を 1回の fetch で把握し、runbook 分岐を固定する

### 5.3 `POST /v1/events/scan|artifact|verify`

追加（202 response = `IngestAccepted`）:

- `endpoint` (string)
- `payload_sha256` (string)
- `accepted_at` (string, RFC3339)

利用目的:

- Gateway 側の ACK 検証
- 監査系で duplicate 判定の追跡容易化

### 5.4 OpenAPI 更新対象

- `docs/openapi/control-plane-v1.yaml`
- `control-plane/api/cmd/sync-openapi-enums/main.go`（enum 同期）

## 7. Exit Criteria

- `zt policy status --json` で `set_consistency` と `freshness_state` が常時取得できる
- `zt sync --json` で backlog 指標 5項目が欠落しない
- ingest 202 応答で `endpoint` / `payload_sha256` / `accepted_at` が必ず返る
- v0.6.0MAX 追加 CI ゲートが 2週間連続で green
- README / OPERATIONS の導線が `error_code -> command -> runbook` の同一順序で記述されている

## 8. Implementation Order (Recommended)

1. Contract test 先行追加（Gateway/CP）
2. CP API schema 追加（policy_set_id, ingest ACK fields）
3. Gateway JSON 出力拡張（policy status / sync --json）
4. OpenAPI 更新 + enum sync
5. CI ゲート追加
6. README / OPERATIONS 更新

## 9. README 更新案（貼り付け用）

挿入位置案: README の「現時点での安全な運用ガイド（推奨）」直下。

追加案:

- v0.6.0MAX では `zt policy status --json` の `set_consistency` / `freshness_state` を運用一次判定の必須項目にする
- `zt sync --json` の `pending_count` / `oldest_pending_age_seconds` を監視し、閾値超過時は `docs/OPERATIONS.md` の backlog runbook を実施する
- Control Plane ingest 202 応答の `payload_sha256` を用いて duplicate/retry を機械判定する
- 設計正本は `docs/architecture/V0.6.0MAX_DESIGN.md`

最短運用コマンド案:

```bash
go run ./gateway/zt setup --json

go run ./gateway/zt policy status --json --kind extension
go run ./gateway/zt policy status --json --kind scan

go run ./gateway/zt send --client <recipient> --share-json ./safe.txt
go run ./gateway/zt sync --json
```
