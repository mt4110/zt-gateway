## 1. Context

`--share-json` は verify command と `receipt_hint` を返せる一方で、Slack/メール投稿本文は連携側で都度組み立てる必要があります。

結果として、運用現場で次のばらつきが発生します。

- 投稿文面のテンプレートがチームごとに分岐する
- verify command は貼られるが receipt command が省略される
- Helpdesk/監査向けの定型文が手運用になる

v0.9.0 では Cloud-Agnostic Wrapper の最小実装として、`--share-json` にチャネル別テンプレートを additive で同梱します。

## 2. Goal / Non-goal

### Goal

- `--share-json` payload に `channel_templates` を追加する
- Slack/メール向けの定型文を `ja` / `en` で返す
- テンプレート内に verify command と receipt command を含める
- 既存フィールド（`kind`/`format`/`command`/`text`/`receipt_hint`）を不変にする

### Non-goal

- `zt send` CLI flag 追加
- `receipt_hint` schema 変更
- 外部 SaaS API への直接投稿機能

## 3. Scope (v0.9.0)

### v0.9.0-1 Share JSON Channel Templates Contract

`zt send --share-json` payload を次のように additive 拡張する。

- `channel_templates` (object)
  - `version` (`v1` fixed)
  - `slack_text` (Slack貼り付け向け本文)
  - `email_subject` (メール件名)
  - `email_body` (メール本文)

### v0.9.0-2 Localization and Command Linkage Contract

- `format=en` のとき英語テンプレートを返す
- `format=ja` のとき日本語テンプレートを返す
- `slack_text` / `email_body` は `command` と `receipt_hint.command` を含む
- `email_subject` は packet basename を含む

### v0.9.0-3 Flow E2E Non-regression Contract

`send -> share-json -> verify -> receipt` の契約を維持しつつ、`channel_templates` が欠落しないことを固定する。

## 5. Compatibility Rule

- `channel_templates` は additive field
- 既存5フィールドの意味は変更しない
- 連携側は未知フィールド無視を継続可能
