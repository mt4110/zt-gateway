## 1. Context

`zt verify --receipt-out` は実装済みだが、送信側 `zt send --share-json` から受信側へレシート出力導線を直接渡せていない。

結果として、現場では次が人手作業になりやすい。

- verify command に `--receipt-out` をどう付けるか
- receipt ファイル名/保存先の命名規則
- CI/Slack 連携で「検証コマンド」と「証跡保存コマンド」を分けて管理する手間

v0.8.0 では `--share-json` を additive に拡張し、既存互換を維持したまま受信レシート導線を機械可読化する。

## 2. Goal / Non-goal

### Goal

- `--share-json` payload に `receipt_hint` を追加する
- `receipt_hint` は deterministic な推奨 path/command を返す
- `send -> share-json -> verify -> receipt` E2E で `receipt_hint` を利用可能にする
- 既存4フィールド（`kind/format/command/text`）の意味を変えない

### Non-goal

- `verify` コマンド仕様の破壊的変更
- receipt schema (`receipt_version=v1`) の更新
- 監査ログ保存先の変更

## 3. Scope (v0.8.0)

### v0.8.0-1 Share JSON Receipt Hint Contract

`zt send --share-json` payload を次のように additive 拡張する。

- `receipt_hint` (object)
  - `version` (`v1` fixed)
  - `path` (suggested relative receipt path)
  - `command` (`zt verify --receipt-out ...` command)

### v0.8.0-2 Deterministic Receipt Path Contract

`receipt_hint.path` は packet basename から deterministic に構築する。

- suffix `.spkg.tgz` を除いた stem を利用
- path token は `[A-Za-z0-9._-]` のみ許可し、その他は `_` へ正規化
- output format: `./receipt_<normalized_stem>.json`

### v0.8.0-3 E2E Receipt Linkage Contract

`send -> share-json -> verify` の contract test で次を固定する。

- `receipt_hint.version == v1`
- `receipt_hint.command` は `command` と同じ artifact を指す
- `receipt_hint.path` を使って `zt verify --receipt-out` したとき receipt が生成される

## 5. Compatibility Rule

- 既存4フィールドは維持（意味変更なし）
- `receipt_hint` は additive field
- 連携側は未知フィールド無視を継続
