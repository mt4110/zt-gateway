# v0.9.2 Design (Draft): Team/Enterprise Boundary Mode

## 0. Goal

`zt-gateway` を「社内/チーム内限定でのみ利用可能」な運用モードへ拡張し、
`v1.0.0` までに enterprise 利用の最小境界制御を成立させる。

## 1. Problem

現行は cryptographic verification は強いが、次の境界制御は運用依存で残る。

- 誰に送ってよいか（recipient boundary）
- 誰の署名を信頼するか（signer boundary）
- どの共有経路を許可するか（channel boundary）

## 2. Scope (v0.9.2)

### C-9201 Recipient Boundary Contract

- `zt send` は team allowlist に含まれる recipient のみ許可
- allowlist 未設定/不整合時は fail-closed

### C-9202 Signer Boundary Contract

- `zt verify` は team signer allowlist と一致した署名者のみ pass
- 未設定時は fail-closed（break-glass は明示フラグのみ）

### C-9203 Channel Boundary Contract

- `--share-route` は enterprise policy の許可経路に限定
- 例: `stdout` のみ許可、`clipboard`/`file:` を禁止可能

### C-9204 Boundary Evidence Contract

- `share-json` / `receipt` / audit event に `tenant_id` と `team_id` を追加
- 監査で「境界内で処理された事実」を再現可能にする

### C-9205 Setup/Doctor Boundary Gate Contract

- `zt setup --json` / `zt config doctor --json` に境界チェック項目を追加
- 主要チェック:
  - boundary policy 読み込み
  - recipient boundary 一致
  - signer boundary 一致
  - disallowed share-route 使用不可

## 3. Policy Shape (Draft)

`policy/team_boundary.toml` を追加する。

```toml
tenant_id = "corp-example"
team_id = "secops"

allowed_recipients = ["clientA", "clientB"]
allowed_signer_fingerprints = ["40HEX...", "40HEX..."]
allowed_share_routes = ["stdout", "none"]

break_glass_allow_degraded = false
break_glass_require_reason = true
```

## 4. CLI Behavior (Draft)

- `zt send`:
  - `--client` が `allowed_recipients` 外なら block
  - `--share-route` が許可外なら block
- `zt verify`:
  - signer fingerprint が `allowed_signer_fingerprints` 外なら block
- `zt setup --json`:
  - `team_boundary_policy`, `team_boundary_recipients`, `team_boundary_signers`, `team_boundary_routes` を checks に追加

## 6. Exit Criteria

- boundary policy 未設定時は send/verify/setup が fail-closed
- allowlist 外 recipient/signer/route が確実に block される
- receipt/share-json/audit に tenant/team 文脈が出力される
- v0.9.2 boundary gate が pre-push で独立検知できる
