# v0.9.2 Design (Draft): Team/Enterprise Boundary

## 0. Baseline

- 確認日: 2026-02-26
- v0.9.1 で成立した cryptographic trust chain を前提に、v0.9.2 は「誰と、どの経路で、どの組織境界内だけで使うか」を fail-closed で固定する
- 対象は `zt` CLI の送信/検証/診断経路と、その監査証跡（share-json/receipt/audit）

## 1. Goal / Non-Goal

### Goal

- tenant/team 単位で recipient/signer/share-route の境界を機械強制する
- 境界設定の欠落・逸脱を `zt setup --json` / `zt config doctor --json` で deterministic に検知する
- 監査時に「境界内で処理した事実」を JSON 証跡から再現可能にする

### Non-Goal

- IdP/SCIM 連携や動的ディレクトリ照会は v0.9.2 範囲外
- マルチ tenant 同居実行（1プロセスで複数 tenant policy 切替）は v0.9.2 範囲外
- ネットワーク境界（FW/Proxy）そのものの管理は本設計の直接対象外

## 2. Contract Scope (C-9201..C-9205)

### C-9201 Recipient Boundary Contract

- `zt send --client <recipient>` は `allowed_recipients` 一致時のみ実行継続
- `allowed_recipients` 未設定、空配列、重複/空文字を含む不正定義は fail-closed
- 不一致時は送信処理（scan/rebuild/pack/share）に入る前に停止し、境界違反として監査イベントに記録する

### C-9202 Signer Boundary Contract

- `zt verify` は verify で得た signer fingerprint が `allowed_signer_fingerprints` と一致した場合のみ pass
- allowlist 未設定または空の場合は fail-closed
- break-glass は明示フラグ + 理由必須（理由未指定は fail-closed）
  - 理由不足は `ZT_SEND_TEAM_BOUNDARY_BREAK_GLASS_REASON_REQUIRED` / `ZT_VERIFY_TEAM_BOUNDARY_BREAK_GLASS_REASON_REQUIRED` で切り分け可能
- break-glass guardrail 強化:
  - `break_glass_require_approver=true` で承認者ID必須
  - `break_glass_max_ttl_minutes>0` で短期 token（`expires_at`）を必須化
  - 期限切れ token は fail-closed

### C-9203 Channel Boundary Contract

- `--share-route` は `allowed_share_routes` のみ許可
- 許可外 route（例: `clipboard`, `file:`）は fail-closed
- route 未指定時に内部既定 route を使う場合も、評価対象は「最終解決 route」とする

### C-9204 Boundary Evidence Contract

- `share-json`, `receipt`, `audit event` に `tenant_id`, `team_id`, `boundary_policy_version` を追加
- 上記 3 系統で同一 `tenant_id/team_id` を保持し、監査時に相互照合できる状態を契約化する
- 境界違反で fail したケースも audit event に残し、成功/失敗の両方で追跡可能にする
- `send/verify` 成功時に audit append が失敗した場合は fail-closed（追跡不能成功を禁止）

### C-9205 Setup/Doctor Boundary Gate Contract

- `zt setup --json` / `zt config doctor --json` の checks に境界専用項目を追加する
- 最低限の追加チェック:
  - `team_boundary_policy_loaded`
  - `team_boundary_recipient_contract`
  - `team_boundary_signer_contract`
  - `team_boundary_share_route_contract`
  - `team_boundary_signer_pin_consistency`（検知コード: `policy_team_boundary_signer_split_brain_detected`）
  - `team_boundary_break_glass_guardrail`（検知コード: `policy_team_boundary_break_glass_guardrail_weak` など）
  - `audit_trail_appendability`（検知コード: `policy_audit_trail_append_unavailable`）
- いずれかが fail の場合、overall を fail-closed 判定に寄せる

## 3. Policy Shape (Draft)

`policy/team_boundary.toml` を新設する。

```toml
tenant_id = "corp-example"
team_id = "secops"
boundary_policy_version = "2026-02-26"

allowed_recipients = ["clientA", "clientB"]
allowed_signer_fingerprints = ["AAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAA", "BBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBB"]
allowed_share_routes = ["stdout", "none"]

break_glass_enabled = false
break_glass_require_reason = true
break_glass_require_approver = true
break_glass_max_ttl_minutes = 60
```

設計上の制約:

- `tenant_id`, `team_id`, `boundary_policy_version` は必須
- `allowed_*` はすべて 1 件以上必須（空配列禁止）
- fingerprint は 40 hex 正規化（大文字/小文字差異を吸収して比較）

## 4. CLI Contract (Draft)

- `zt send`:
  - `--client` を recipient boundary で検査
  - 解決後 `share-route` を channel boundary で検査
  - 境界違反時は pack/share 実行前に停止
- `zt verify`:
  - verify 成功後の signer fingerprint を signer boundary で検査
  - 不一致は verify overall fail
- `zt setup --json` / `zt config doctor --json`:
  - 境界チェック結果を `checks[]` に追加
  - 境界 policy 欠落時は remediation ヒント付きで fail

## 5. Implementation Tickets (TKT-9201..TKT-9205)

## TKT-9201 Team Boundary Policy Loader

- 変更対象:
  - `gateway/zt` の policy load / validation / setup / doctor 周辺
  - `policy/team_boundary.toml`（新規）
- DoD:
  - TOML 読み込み + 構造検証 + fail-closed エラー分類が成立
  - `zt setup --json` / `zt config doctor --json` から境界 policy の有効性が見える

## TKT-9202 Send Path Boundary Enforcement

- 変更対象:
  - `gateway/zt/commands_flow.go`
  - `gateway/zt/commands_share.go`
  - 送信経路の JSON 出力契約テスト
- DoD:
  - recipient/share-route 境界が送信前に強制される
  - 許可外 client/route は必ず block される

## TKT-9203 Verify Signer Boundary Enforcement

- 変更対象:
  - `gateway/zt/commands_verify.go`
  - `tools/secure-pack` 側 signer fingerprint 受け渡し経路
- DoD:
  - verify 実測 signer と allowlist を照合できる
  - allowlist 不一致・未設定時に fail-closed で止まる

## TKT-9204 Boundary Evidence Fields

- 変更対象:
  - `gateway/zt/verify_receipt.go`
  - `gateway/zt/events_audit.go`
  - `gateway/zt/share_transport.go` または share-json 生成経路
- DoD:
  - share-json/receipt/audit の 3 系統で tenant/team/version が整合
  - 境界違反ケースの失敗監査イベントが残る

## TKT-9205 Boundary CI Gate

- 追加:
  - `scripts/ci/check-v092-boundary-gate.sh`
- 更新:
  - `scripts/ci/check-pre-push-readiness.sh`
- DoD:
  - v0.9.2 境界契約だけを独立実行できるゲートがある
  - pre-push で境界回帰を他ゲートと分離して検知できる

## 6. Recommended Order

1. TKT-9201 Policy Loader
2. TKT-9202 Send Enforcement
3. TKT-9203 Verify Enforcement
4. TKT-9204 Boundary Evidence
5. TKT-9205 CI Gate

## 7. Exit Criteria

- C-9201..C-9205 を満たす自動テスト/契約テストが揃っている
- boundary policy 欠落・不正・空 allowlist で `send/verify/setup/doctor` が fail-closed する
- allowlist 外 recipient/signer/share-route が deterministic に block される
- share-json/receipt/audit で tenant/team/version が一貫して出力され、監査で相互照合できる
- `check-v092-boundary-gate.sh` が pre-push に統合され、境界回帰を独立検知できる
