# v0.9.2 Abnormal Use Cases / Incident Runbook

`Team Boundary` を有効化した運用で、利用不能を最短で復旧するための異常系ユースケース集です。  
対象: `zt send` / `zt verify` / `team_boundary.toml` / signer key 運用。

## 0. 使い方

1. まず `ZT_ERROR_CODE` と `SECURE_PACK_ERROR_CODE` を採取する
2. 本文の「異常系マップ」で該当ケースを特定する
3. 対応 runbook を実行し、Exit Criteria を満たしたら復旧完了とする

## 1. 異常系マップ（v0.9.2）

| ケース | 代表エラーコード | 影響 | 最優先アクション |
| --- | --- | --- | --- |
| team boundary policy 欠落/破損 | `ZT_SEND_TEAM_BOUNDARY_POLICY_FAILED`, `ZT_VERIFY_TEAM_BOUNDARY_POLICY_FAILED` | send/verify 停止 | `policy/team_boundary.toml` を復旧し構文検証 |
| recipient 境界違反 | `ZT_SEND_TEAM_BOUNDARY_RECIPIENT_DENIED` | 特定宛先のみ送信不能 | recipient allowlist を修正、緊急時のみ break-glass |
| share-route 境界違反 | `ZT_SEND_TEAM_BOUNDARY_SHARE_ROUTE_DENIED` | 共有導線停止 | `allowed_share_routes` と実運用 route を一致させる |
| signer 境界違反 | `ZT_VERIFY_TEAM_BOUNDARY_SIGNER_DENIED` | verify 停止 | signer allowlist と team boundary signer を整合 |
| signer allowlist 未設定 | `ZT_VERIFY_SIGNER_PIN_MISSING`, `SP_SIGNER_PIN_MISSING` | verify 全停止 | signer allowlist を即時配布 |
| signer allowlist 不一致 | `ZT_VERIFY_SIGNER_PIN_MISMATCH`, `SP_SIGNER_PIN_MISMATCH` | verify 停止 | 鍵ローテ/喪失を疑い runbook 実行 |
| signer allowlist 設定不正 | `ZT_VERIFY_SIGNER_PIN_CONFIG_INVALID`, `SP_SIGNER_PIN_CONFIG_INVALID` | verify 停止 | fingerprint 形式（40/64 hex）を修正 |
| signer policy split-brain | `policy_team_boundary_signer_split_brain_detected` (`team_boundary_signer_pin_consistency`) | 端末/環境ごとに verify 成否が分岐 | signer pin ソースと `team_boundary.toml` を同一化 |
| break-glass guardrail 弱い | `policy_team_boundary_break_glass_guardrail_weak` (`team_boundary_break_glass_guardrail`) | 誤運用で恒常 override になりやすい | approver必須 + TTL必須へ修正 |
| break-glass override 戻し忘れ | `ZT_SEND_TEAM_BOUNDARY_BREAK_GLASS_ENV_PRESENT`, `ZT_VERIFY_TEAM_BOUNDARY_BREAK_GLASS_ENV_PRESENT`, `policy_team_boundary_break_glass_env_present` (`team_boundary_break_glass_guardrail`) | 常時 override で境界逸脱リスク増大 | `ZT_BREAK_GLASS_REASON` を即時除去 |
| break-glass token 形式不正 | `policy_team_boundary_break_glass_token_invalid` (`team_boundary_break_glass_guardrail`) | 緊急時 override 不能 | reason token 形式を修正 |
| break-glass token 期限切れ | `policy_team_boundary_break_glass_token_expired` (`team_boundary_break_glass_guardrail`) | 緊急時 override 不能 | 新しい短期 token を再発行 |
| 監査書き込み失敗（追跡不可） | `ZT_SEND_AUDIT_APPEND_FAILED`, `ZT_VERIFY_AUDIT_APPEND_FAILED`, `policy_audit_trail_append_unavailable` | 成功処理の追跡不能（fail-closed） | spool/audit path 修復後に再実行 |
| break-glass 理由不足 | `ZT_SEND_TEAM_BOUNDARY_BREAK_GLASS_REASON_REQUIRED`, `ZT_VERIFY_TEAM_BOUNDARY_BREAK_GLASS_REASON_REQUIRED`, `policy_team_boundary_break_glass_reason_required` | 緊急回避不能 | `--break-glass-reason` を指定 |

## 2. Runbook: 署名鍵喪失（PC故障・鍵紛失）

### トリガ

- `ZT_VERIFY_SIGNER_PIN_MISMATCH` / `SP_SIGNER_PIN_MISMATCH`
- 送信側 PC の秘密鍵が復元不能

### 手順

1. 影響範囲を固定する  
送信停止を宣言し、未検証ファイルを配布しない。
2. 新署名鍵を発行する  
隔離端末で新鍵を生成し fingerprint を確定する。
3. allowlist を二重窓で更新する  
`tools/secure-pack/SIGNERS_ALLOWLIST.txt`（または `ZT_SECURE_PACK_SIGNER_FINGERPRINTS`）へ旧+新を併記。
4. team boundary も同時更新する  
`policy/team_boundary.toml` の `allowed_signer_fingerprints` に旧+新を併記。
5. 検証テストを実行する  
既存packet（旧鍵）と新規packet（新鍵）の両方で `zt verify` を通す。
6. 周知後に旧鍵を除去する  
全端末反映確認後、旧 fingerprint を allowlist から削除。

### Exit Criteria

- 主要端末で `zt verify` 成功率が復帰
- `SP_SIGNER_PIN_MISMATCH` が監視上で収束
- 旧鍵除去後も新鍵 packet が安定検証可能

## 3. Runbook: 誤ブロック解除（境界設定ミス）

### トリガ

- 実運用で許可されるべき送受信が boundary で block
- 代表: `ZT_SEND_TEAM_BOUNDARY_RECIPIENT_DENIED`, `ZT_SEND_TEAM_BOUNDARY_SHARE_ROUTE_DENIED`, `ZT_VERIFY_TEAM_BOUNDARY_SIGNER_DENIED`

### 手順

1. block の種類を確定する  
recipient / route / signer のどれで止まったかをログで分類。
2. 設定値と実値を比較する  
`team_boundary.toml` の allowlist と CLI 実引数（最終 share-route 含む）を照合。
3. 緊急時は break-glass を短時間で許可する  
`--break-glass-reason "incident=<id>;approved_by=<id>;expires_at=<RFC3339>"` を付与して一時運用。
4. 恒久修正を反映する  
`team_boundary.toml` を修正し、レビュー後に配布。
5. break-glass を必ず解除する  
恒久修正適用後は通常モードへ戻す。

### Exit Criteria

- 同ユースケースで break-glass なしに送受信成功
- 監査ログに incident-id と修正時刻が残る

## 4. Runbook: 計画的署名鍵ローテーション

### 目的

停止を出さずに鍵を更新する（ゼロダウンタイム）。

### 手順

1. 事前準備  
新鍵を発行し fingerprint を確定、ローテ日程を確定。
2. 配布フェーズ（受信側先行）  
allowlist と `team_boundary.toml` に旧+新を追加して先に全端末へ配布。
3. 切替フェーズ（送信側）  
送信側の署名鍵を新鍵へ切替。
4. 監視フェーズ  
`SP_SIGNER_PIN_MISMATCH` が増えていないことを確認。
5. 収束フェーズ  
周知済み猶予期間後に旧鍵 fingerprint を削除。

### Exit Criteria

- 切替期間中に verify 停止インシデントが発生しない
- 旧鍵削除後も verify 成功率維持

## 5. Runbook: Policy配布ずれ（Signer Split-Brain）

### Signer Policy Split-Brain Detected

### トリガ

- `zt config doctor --json` の `checks[].name=team_boundary_signer_pin_consistency` が `fail`
- `checks[].code=policy_team_boundary_signer_split_brain_detected`
- 同じ packet が端末Aでは `ZT_VERIFY_SIGNER_PIN_MISMATCH`、端末Bでは成功する

### 手順

1. 現在の実効 pin ソースを固定する  
`team_boundary_signer_pin_consistency` の message から `source=`（env/file）を確認する。
2. 正本を決める  
原則は `policy/team_boundary.toml` の `allowed_signer_fingerprints` を正本にする（緊急ローテ中の二重窓を除く）。
3. verify pin ソースを同期する  
`ZT_SECURE_PACK_SIGNER_FINGERPRINTS` / `SECURE_PACK_SIGNER_FINGERPRINTS` / `SIGNERS_ALLOWLIST.txt` のいずれか運用経路を1つに統一し、正本と同じ fingerprint 集合にする。
4. 全端末で doctor を再実行する  
`zt config doctor --json` で `team_boundary_signer_pin_consistency=ok` を確認する。
5. 実 packet で verify を確認する  
代表 packet で端末横断の verify 結果が一致することを確認する。

### Exit Criteria

- 全運用端末で `team_boundary_signer_pin_consistency` が `ok`
- boundary policy と verify pin の集合差分（missing/extra）が 0
- 同一 packet の verify 成否が端末間で一致する

## 6. Runbook: Break-Glass 戻し忘れ/濫用

### Break-Glass Override Left Enabled

### トリガ

- `zt config doctor --json` の `checks[].name=team_boundary_break_glass_guardrail` が `fail`
- `checks[].code` が以下のいずれか:
  - `policy_team_boundary_break_glass_guardrail_weak`
  - `policy_team_boundary_break_glass_env_present`
  - `policy_team_boundary_break_glass_token_invalid`
  - `policy_team_boundary_break_glass_token_expired`
- `zt send` の `ZT_SEND_TEAM_BOUNDARY_BREAK_GLASS_REASON_REQUIRED`
- `zt verify` の `ZT_VERIFY_TEAM_BOUNDARY_BREAK_GLASS_REASON_REQUIRED`

### 手順

1. 常時 override を除去する  
`ZT_BREAK_GLASS_REASON` を shell/profile/CI から削除し、常駐設定を禁止する。
2. guardrail を strict 化する  
`team_boundary.toml` で `break_glass_require_reason=true`, `break_glass_require_approver=true`, `break_glass_max_ttl_minutes=<short>` を設定する。
3. reason token を標準化する  
緊急時のみ `incident=...;approved_by=...;expires_at=<RFC3339>` 形式を発行し、短期間で失効させる。
4. doctor で再検証する  
`zt config doctor --json` で `team_boundary_break_glass_guardrail=ok` を確認する。
5. 実運用導線を再確認する  
break-glass 実行後に通常モードへ戻る運用手順（解除手順）を runbook と当番手順へ反映する。

### Exit Criteria

- `team_boundary_break_glass_guardrail` が全端末で `ok`
- `ZT_BREAK_GLASS_REASON` の常駐設定が 0
- break-glass 実行は期限付き token + 承認者ID でのみ実行される

## 7. Runbook: 監査書き込み失敗（追跡不可）

### Audit Trail Append Failed

### トリガ

- `zt send` 成功直前で `ZT_SEND_AUDIT_APPEND_FAILED`
- `zt verify` 成功直前で `ZT_VERIFY_AUDIT_APPEND_FAILED`
- `zt config doctor --json` の `checks[].name=audit_trail_appendability` が `fail`
- `checks[].code=policy_audit_trail_append_unavailable`

### 手順

1. spool 書き込み可否を確認する  
`ZT_EVENT_SPOOL_DIR`（未設定時 `.zt-spool`）の権限/所有者/容量を確認する。
2. audit チェーン破損を確認する  
`zt audit verify --file .zt-spool/events.jsonl` を実行し、破損行があれば backup のうえ修復する。
3. lock 残骸を解消する  
異常終了で残った `.zt-spool/.lock` を確認し、実プロセス不在を確認してから除去する。
4. doctor で追記可能性を確認する  
`zt config doctor --json` で `audit_trail_appendability=ok` を確認する。
5. 失敗した送受信を再実行する  
追跡不能で fail-closed した操作を再実行し、監査ログへの追記を確認する。

### Exit Criteria

- `audit_trail_appendability` が全運用端末で `ok`
- `zt send` / `zt verify` が監査追記つきで正常終了する
- `events.jsonl` チェーン検証が `zt audit verify` で pass する

## 8. v1.0.0 Local Web ダッシュボード要件（推奨）

`tracer dashboard` / `ファイル管理画面` で最低限見えるべき項目:

- `event_id`, `occurred_at`, `target_name`, `recipient_name`
- `tenant_id`, `team_id`, `boundary_policy_version`
- `policy_decision.reason_code`, `policy_decision.error_code`
- `secure_pack_error_code`（存在時）
- `actual_signer_fingerprint`, `allowed_signer_fingerprints`（不一致時）
- `boundary_break_glass`, `boundary_break_glass_reason`

推奨ビュー:

1. 失敗率トレンド（error_code 単位）
2. signer mismatch 一覧（ファイル単位）
3. break-glass 実行監査（理由・実行者・時刻）

## 9. 四半期訓練チェックリスト（鍵喪失 / 誤ブロック）

四半期ごとに最低 1 回、以下 2 シナリオを演習する。

| シナリオ | 目的 | 成功条件 |
| --- | --- | --- |
| 鍵喪失訓練（Signer Key Loss） | 鍵紛失時に verify 全停止を最短復旧できることを確認 | Runbook 2 を 60 分以内に完走し、旧+新鍵で `zt verify` 成功 |
| 誤ブロック訓練（False Block） | boundary 設定ミス時に安全に解除し恒久修正へ戻せることを確認 | break-glass 理由付き一時運用から恒久修正へ戻し、再発ケースで break-glass 不要 |

実施チェック（四半期）:

- 連絡網: インシデント司令、承認者、実行者を当日体制へ更新
- 設定資材: `team_boundary.toml` と signer allowlist の最新版を訓練環境へ反映
- 鍵喪失演習: `SP_SIGNER_PIN_MISMATCH` を意図発生させ、Runbook 2 の全手順を記録
- 誤ブロック演習: recipient/route/signer のいずれかを意図誤設定し、Runbook 3 を記録
- 監査確認: `boundary_break_glass_reason` と復旧時刻が監査ログに残ることを確認
- 事後改善: 実測 RTO、詰まりポイント、手順改訂点を 1 ページで残す
