## 市場文脈とPMF仮説

### 類似カテゴリ

- Wormhole
- Proton Drive sharing
- Bitwarden Send
- Tailscale Taildrop
- Tresorit
- OnionShare

### PMF仮説

`zt-gateway` は次の3要素を同時に満たすと選ばれる。

1. ローカルファーストで検証可能な安全性
2. ゼロ設定に近い転送UX
3. 既存業務導線（Slack/メール/ファイルドロップ）への自然統合

## 差別化優先順位（重要順）

1. ゼロ設定体験
2. 既存業務導線への埋め込み
3. 安全性の見える化（1行で状態提示）
4. 運用者価値（監査証跡・ポリシー配布・自己修復ガイド）
5. 互換性吸収（OS/実行環境差分の自動吸収）

## ロードマップ

## `v0.3` UX基盤（無意識利用の初手）

### 到達目標

セキュリティ知識が少なくても、送信/検証を短時間で完了できる状態を作る。

### マイルストーン

- Quick Startを製品UXの主導線として固定（送信3分 / 受信1分）
- `zt setup --json` を起点にしたガイド付き初期化と修復提案
- 共有導線の固定化（`--share-json` 契約の安定運用）
- pin mismatchのUX化:
  - 平易な原因説明
  - 実行環境別のワンコマンド修正候補提示
- Helpdesk向けエラー分類の標準化

### Exit Criteria

- 初回送信成功率（内部検証） `>= 80%`
- Time to First Send 中央値 `<= 5分`
- 主要セットアップ失敗上位3件に決定論的Quick Fixを提供

## `v0.5` 導線統合と運用価値

### 到達目標

普段の業務導線で自然に使われ、運用負荷を下げながら拡張できる状態を作る。

### マイルストーン

- 導線アダプタ整備:
  - Slack投稿テンプレート
  - メール共有テンプレート
  - command-file handoffパターン
- ポリシー配布経路:
  - 署名付きpolicy bundle
  - 段階展開（canary -> default）
- 監査証跡MVP:
  - send/verifyイベントログスキーマ
  - 改ざん検知可能な追記戦略
- CLI出力から自己修復runbookへ直リンク
- 互換性リゾルバ:
  - `env` / `built-in` / 併用状態の明示
  - OS・配布経路別の修正案提示

### Exit Criteria

- 実運用導線2系統（例: Slack + メール）を継続運用
- 転送あたりの問い合わせ件数を `v0.3` 比で `40%` 削減
- 本番sendイベントの `100%` が監査ログ対象

## `v1.0` 製品化とカテゴリ差別化

### 到達目標

検証可能な安全性を摩擦なく提供する、標準的なセキュア転送レイヤとして確立する。

### マイルストーン

- install-and-use体験の完成:
  - 標準導線で鍵/環境変数の手動説明を必須にしない
  - 送受信とも安全なデフォルトを適用
- Trust Status Lineの完成:
  - 主要導線すべてで1行判定（検証済み/改ざんなし/ポリシー準拠）
- エンタープライズ運用パッケージ:
  - policy governance playbook
  - 鍵ローテーション運用パッケージ
  - インシデント対応runbook統合
- 互換性保証マトリクス:
  - 主要OS/実行環境差分を検証済みにする
- GA移行パッケージ:
  - 1.0判定基準
  - pre-1.0からの移行ガイド

### Exit Criteria

- 初回実行成功率 `>= 90%`
- 受信側検証完了率 `>= 95%`
- 転送フローのP1障害 `30日連続ゼロ`

## 実装順（実行シーケンス）

1. UX計測基盤の確立（`v0.3` week 1）
2. Guided setup + pin mismatch UX（`v0.3` week 1-3）
3. 共有契約の硬化 + 導線テンプレート（`v0.3` week 2-4）
4. 導線アダプタ + 監査ログスキーマ（`v0.5` 前半）
5. ポリシー配布と段階展開（`v0.5` 中盤）
6. 自己修復ガイドと互換性リゾルバ（`v0.5` 後半）
7. Trust Status Line統一 + 運用パッケージ（`v1.0`）
8. GAゲート判定、移行資料整備、最終リリース判定

## 付箋用サマリー

- `v0.3`: 初回利用を迷わせない。失敗は必ず自己修復へ導く。
- `v0.5`: 既存業務導線に溶け込ませ、運用負荷を下げる。
- `v1.0`: 検証可能な安全性を、無意識の標準体験にする。

## 主なリスクと対策

- リスク: UX簡素化で安全前提が見えなくなる
  - 対策: fail-closed維持 + Trust Status Line常時表示
- リスク: 環境差分によるpin不整合が再発する
  - 対策: 互換性リゾルバ + 環境別修正案の自動提示
- リスク: 連携機能の増加で保守面積が広がる
  - 対策: `--share-json` 固定契約 + 最小アダプタ方針

## 進行管理

- 毎週: ロードマップ定例（進捗、阻害要因、KPIトレンド）
- RCごと: Exit Criteria判定（`v0.3` / `v0.5` / `v1.0`）
- 障害後: 1スプリント以内にrunbookとUX文言へ反映
